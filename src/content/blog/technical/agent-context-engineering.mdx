From 2023 to 2025, prompt engineering was the focus in applied AI. Starting last year, the term context engineering came to the foreground. Building with language models is becoming less about finding the right words and phrases for the instructions, and more about answering the broader question of "does the model have everything it needs to complete the task?"

Within a session, AI agents are capable of managing some of its own context through context compression and memory maintenance. Claude Code, for example, will compact its context window when it fills up. But what agents can't manage is the quality of the information external to themselves: the file system, the knowledge base, the tool descriptions, the docs, the examples, the constraints. That part remains a human responsibility, and it's where most of the real work lives.

Shopify's CEO Tobi Lütke and AI researcher Andrej Karpathy tweeted in mid-2025. "I really like the term 'context engineering' over prompt engineering," wrote Tobi. Karpathy agreed, noting that people associate prompts with short instructions, whereas in every serious LLM application, context engineering is "the delicate art and science of filling the context window with just the right information for each step."

## Context Engineering vs. Prompt Engineering

Prompt engineering mattered more when models were worse at understanding intent. The early practice of carefully phrasing instructions — specific role definitions, elaborate chain-of-thought triggers, examples to get the desired format — was a workaround for models that needed heavy guidance to stay on track. Current state-of-the-art LLMs have largely solved the "what should I do" problem. They follow instructions well. You don't need to spend much time engineering the prompt itself.

What they can't solve on their own is the "how we do it _here_" problem. Models trained on public data know how the world generally works, but not how your organization or your product specifically operates. Context engineering is the work of making that private, product-specific knowledge available to the agent at the right time.

Andrej Karpathy's analogy is useful here: think of the LLM as a CPU and its context window as RAM. The CPU can only operate on what's in RAM. It doesn't matter how capable the processor is if the wrong data is loaded. Context engineering is the discipline of deciding what gets loaded, and ensuring it's accurate, current, and well-structured enough to be useful.

For agents specifically, the problem of wrong context compounds in a way it doesn't for single-turn assistants. Each subsequent step in an agent's execution uses the outputs of prior steps as inputs. A stale policy retrieved in step two becomes the assumption underlying steps three, four, and five. By the time the agent produces a final answer, it may be confidently wrong, which is worse than saying I don't know. And this wouldn't be because of a reasoning failure, but because the information it reasoned from was bad from the start.

## Most Agent Failures Are Context Failures

The main thing that determines whether an agent succeeds or fails is the quality of the context you give it. Most agent failures are not model reasoning failures — they are context failures. The failure modes are well-documented:

**Stale knowledge.** RAG systems have no native awareness of whether retrieved documents are current. Outdated documents can score just as high on semantic similarity as fresh ones, so a retriever has no basis for preferring the accurate version. The result is that a knowledge base that was correct at launch silently drifts: a customer support agent still references an old refund policy, a pricing agent quotes last quarter's rates, a coding agent recommends a deprecated API pattern. The model doesn't know it's wrong; it just answers confidently from what it finds.

**The "lost in the middle" problem.** [Research from Stanford published in the ACL Transactions journal](https://arxiv.org/abs/2307.03172) found that model performance follows a U-shaped curve as context grows: accuracy is highest when relevant information appears at the beginning or end of the context window, and drops significantly when it's in the middle. In multi-document QA tasks, moving the answer from position 1 to position 10 in a 20-document context caused a 30-point accuracy drop. Bigger context windows delay the onset of this problem but don't eliminate it — they just push the degradation further out.

**Contradictory retrieval.** When a knowledge base contains both current and outdated information on the same topic, a retriever may surface both, and the model has to choose — often arbitrarily. [A systematic evaluation of RAG systems in healthcare](https://arxiv.org/abs/2511.06668) found that the presence of contradictory documents caused a measurable drop in answer quality across thousands of test cases. In production, this typically shows up as an agent that gives inconsistent answers to the same question depending on which documents happened to rank highest that day.

**Tool description quality.** [Recent research testing 17 models](https://arxiv.org/abs/2505.18135) found that simply editing a tool's description — adding clearer language about when to use it, or examples of appropriate usage — caused agents to use that tool more than ten times as often. The inverse is also true: vague or overlapping tool descriptions cause agents to select the wrong tool, produce unnecessary API calls, and pad the context window with irrelevant output. Tool descriptions are not documentation; they are load-bearing context that shapes agent behavior at runtime.

## A Four-Layer Framework for Agent Context

To reason about agent context systematically, it helps to decompose it into four layers.

### Layer 1: Instruction Context

This is the layer most teams invest in first: system prompts, goals, rules, policies, persona definitions, and behavioral constraints. Few-shot examples belong here too — rather than trying to enumerate every possible edge case in the prompt, the more effective approach is to provide a set of diverse, representative examples that demonstrate the expected behavior. A long laundry list of rules is harder for a model to apply consistently than a handful of well-chosen examples.

The instruction layer sets the agent's identity and operating boundaries, but it's only one part of the picture.

### Layer 2: Knowledge Context

This is where documentation, examples, retrieved content, and domain knowledge live. It's the layer most directly affected by retrieval pipelines and knowledge base quality, and it's where the difference between a demo and a production system usually lives — not in which model you're using, but in how carefully information is selected, structured, and delivered at each step.

Knowledge context is also the layer most vulnerable to staleness. Instruction files, knowledge bases, and retrieval stores are snapshots of what was true at the time they were written. They don't update themselves when an upstream product ships a breaking change. Without active maintenance, they accumulate inaccuracies silently.

### Layer 3: Operational Context

This is the runtime state: the current conversation history, prior tool outputs, the user's immediate request, and environmental signals like timestamps or session metadata. A context window flooded with irrelevant logs, stale tool outputs, or accumulated history can distract the model, causing it to fixate on earlier patterns rather than the immediate instruction.

Managing operational context means deciding what to keep, what to summarize, and what to drop. Long-running agents accumulate a large number of tokens across LLM calls and tool calls — left unmanaged, this balloons cost, increases latency, and degrades performance.

### Layer 4: Action Context

This layer comprises the tools available to the agent — their schemas, descriptions, permissions, and affordances. If a human engineer can't definitively say which tool should be used in a given situation, an agent can't be expected to do better. Bloated tool sets and ambiguous descriptions are among the most common sources of agent misbehavior in production.

Action context also includes what the agent is *not* allowed to do — the permission boundaries and escalation policies that prevent an agent from taking actions that are technically available but outside acceptable scope. Defining these constraints explicitly is as important as defining the tools themselves.

## What You Actually Control

There's a quiet misnomer in "context engineering" that becomes obvious once you deploy against a closed-source model.

With models like Claude, GPT-4o, or Gemini, you do not actually decide which piece of context the agent retrieves, attends to, or weighs at each step. The model does. You can shape the retrieval pipeline — what goes into your vector store, how chunks are sized, what metadata is attached — but the moment the agent is running, the model determines what it finds relevant. You are not engineering the context. You are engineering the environment the agent operates inside.

This distinction matters practically. Teams who think of themselves as "doing context engineering" often focus on prompt structure, retrieval tuning, and token budgeting. Those things matter. But the lever you actually have the most control over — and that has the most impact on output quality — is the accuracy and structure of the underlying information the agent can access. If your docs are wrong, stale, or inconsistently structured, no amount of retrieval tuning fixes it. The model will confidently surface the wrong answer, because that is what it found.

Context engineering, properly understood, is less about controlling what goes into the context window at runtime and more about maintaining the information environment that the agent draws from. The question stops being "how do I get the right chunk into the context window?" and becomes "is everything in my knowledge layer accurate enough that it doesn't matter which chunk the model retrieves?"

## Context Engineering Is a Living Discipline

Context engineering is not a one-time setup. Agents degrade as the world changes — APIs update, products ship new features, policies are revised, support teams discover new edge cases — and unless context is actively maintained, the agent silently becomes worse.

Teams tend to build knowledge systems as one-time projects: ingest documents, deploy, done. Nobody owns ongoing updates. Sources change silently. Policy documents, product specs, and internal guides update regularly, but the AI's knowledge doesn't, because there's no automated sync. The result is content drift: a knowledge base that was accurate at launch and quietly wrong six months later, with no visible signal that anything has changed.

The software engineering discipline has spent decades building tools to manage change in code — version control, dependency management, continuous integration, automated testing. Agent knowledge needs the same rigor, not because it's glamorous work, but because the alternative is agents that slowly get worse at their jobs while everyone assumes they're fine.
