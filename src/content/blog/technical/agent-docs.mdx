Alas. Independent of llms.txt, it's undeniable that more agents are accessing docs, and this trend will likely continue. This article tries to outline what adjustments are worth making based on LLM research from frontier labs. To be clear, we're not advocating for a separate set of "agent docs". ## How agents actually consume your docs today

Before we talk about optimization, it's worth understanding the concrete ways agents encounter your documentation. There are 4 main pathways, and you can decide based on your own product which ones to optimize for.

### 1. RAG systems (support bots, "Ask AI" features)

Tools like Kapa, Intercom's Fin, doc platform’s “Ask AI” feature, and in-house support chatbots are all in this category. They ingest your docs, split them into chunks, and retrieve relevant chunks when a user asks a question.

The chunking typically happens at heading boundaries. So the text under an `## H2` or `### H3` heading becomes one retrievable unit. And imagine having 10 of these units coming from potentially very different places in the docs all fed into the context window. That’s what the LLM will see.

### 2. File system - monorepo’s `/docs` directory and AGENTS.md

Many coding agents consume documentation directly from the repo filesystem. When an agent runs inside a workspace (e.g., in Cursor, Claude Code, or Codex), it can read files such as `README.md`, `docs/`, `CONTRIBUTING.md`, or specialized files like `AGENTS.md`.

Some studies like this [one](https://arxiv.org/pdf/2601.20404) argues that having repository-level context files such as `AGENTS.md` is strictly better, but their sample size is too small IMO (124 pull requests from 10 repos). A more nuanced view from a 2026 study ([Gloaguen et al. 2026](https://arxiv.org/html/2602.11988v1)) had more actionable findings:
- **LLM-generated** context files reduced performance by about 3% on average while increasing cost by 20%.
- **Developer-written** context files, by contrast, actually helped. The paper reports a +4% average improvement in task success rate.
- The reason developer-written files helped is that they contained information that wasn't already in the repository, eg. tooling preferences, workflow requirements, conventions that existed in developers' heads but not in any documentation.

So the takeaway here is that, if you want to write AGENTS.md, don’t be lazy 😛. But since the performance improvements are in the low single digits, we’ll still focus on improvements to human-facing docs.

### 3. MCP servers

The [Model Context Protocol](https://modelcontextprotocol.io/) is an open standard that lets agents programmatically interact with external data sources including documentation. Under the hood, some of these MCP servers use chunking and indexing similar to RAG systems, and some do keyword or fuzzy keyword search. Tool descriptions and schemas are much more important here. This is again out of the scope of this blogpost, but geek out with me here for a sec - [Trilcke et al. 2025](https://arxiv.org/abs/2508.13774) qualitatively states that docstrings make MCP tool choices better and more stable. (Sometimes docstrings fall into tech writer’s purview, sometimes engineering’s purview, but regardless, topic for another day.)

### 4. Web search and crawling

When agents like ChatGPT, Perplexity, and Claude do web browsing, it finds your docs the same way search engines like Google do - through indexable pages, site structure, and links. 
Once it gets a list of results from the search, it fetches the raw HTML (or rendered text) from those pages. The main difference from RAG systems is that the retrievable unit is usually pages instead of chunks.

This is the recommendation we feel most strongly about, and it's one that many technical writers already follow as a matter of good practice.

When an agent retrieves a chunk of your docs, it gets that chunk void of surrounding context. If the chunk says "using the payment object from above," the agent has no idea what "above" refers to. Some agents will attempt to find the reference, but most RAG systems only allow vector search instead of directly allowing agents to retrieve another chunk above it.

There’s overwhelming evidence in LLM research of “reward hacking”, meaning the model taking shortcuts to pass the evaluation without completing the real objective. Coupled with the fact that most agents have a `max_turns` parameter set to a non infinite value, this can mean that if you have lots of references like “above”, “the previous page” etc, agents might end up wasting a lot of turns following the references and end up not exploring enough context that actually matter.

Another piece of evidence comes from [Levy et al. 2025](https://arxiv.org/abs/2503.04388). The researchers kept the context length and position of relevant information constant while varying the number of documents, and find that increasing the document count in RAG settings can reduce performance by up to 20%

> write as if you're doing a radio interview and you don't want to be quoted out of context.  

-– November 2025 Write the Docs [Newsletter](https://www.writethedocs.org/blog/newsletter-november-2025/#optimizing-docs-for-llms)

**What to do:** Avoid forward and backward references like "see above," "as described in the previous section," or "same as before." Instead, restate the key information. This doesn't mean duplicating entire sections, just the critical bits that make a section more understandable on its own.

Many style guides already discourage "see above" and similar phrases, this is just a stronger version of that existing practice because it matters even more for agents. This applies to code snippets as well. If you can, try to make each snippet make sense on its own.

</div>

This is the recommendation where optimizing for agents and optimizing for humans diverge the most. But the form factor doesn’t have to conflict too much. You can add these content as additional expandable sections at the bottom of the page. Or if it makes the page too long, you can put them in a dedicated section of docs structure (think an additional section in the sidebar). 

Research on coding agents ([Dong et al. 2025](https://arxiv.org/pdf/2508.00083)) found that agents struggle the most with "unwritten team conventions" and "non-public, highly contextualized information." And research on agent self-recovery ([Del Rosario et al. 2025](https://arxiv.org/abs/2509.08646)) found that agents with access to structured failure information performed significantly better than agents reasoning about errors from their general knowledge. (Side note: this author believes agent self-recovery and self-evolution will become even more important in the future!)

A recent paper studied automated prompt optimization for LLM agents ([Brookes et al. 2025](https://arxiv.org/abs/2512.09108)). They found that while many human prompts start with vague guidance like “consider edge cases” or “be thorough”. Afterwards the LLM decided to optimize the prompts by rewriting these into explicit, structured instructions and example edge cases, the accuracy improved by 10.1%–36.9%.

Stripe does it well with documenting failure modes. Its error codes page maps each code to a description, and some of them with resolution steps. The API also returns a `doc_url` field pointing directly to the relevant doc entry.

**What to do:** The lower hanging fruit is adding dedicated debugging/error sections. Try to step into the agent's shoes who only have a vague (often wrong) impression of your product but is now expected to recite your API docs from memory. You can either distribute them to each page (put those expandable components to work) or have dedicated sections per main product area. 

If you want to go a step further, you can also add edge cases and known issues. Typically, you’ll find known bugs in github issues, changelogs, release notes, etc.. And edge cases normally live in support conversations. You can give LLM links and files to find the most common patterns and synthesize them into docs (or you can try Promptless 😄).

</div>

Human docs typically show one or two examples per endpoint. Research on LLM performance shows significant gains when more examples (even as many as 10+) are provided ([Agarwal et al. 2024](https://arxiv.org/abs/2404.11018)). Interestingly, LLM also benefits from negative examples more often than humans do. Giving it "don't do this" examples are particularly effective ([Mo et al. 2024](https://arxiv.org/abs/2402.11254)).

That said, this is also just good documentation practice. More examples help humans too, especially when they cover common use cases. One downside is that they're hard to maintain. (wink wink)

**What to do:** For your most critical endpoints, consider expanding the number of examples and add negative examples on what agents shouldn’t do. Try to minimize repetitiveness.

There are not many people doing this in production yet, but we see the top dev tools trending in this direction. Some have example directories in their github repo like [Vercel](https://github.com/vercel/vercel/tree/main/examples), some have cookbooks like [OpenAI](https://github.com/openai/openai-cookbook) (this author has found them both very helpful). Publishing these examples in github in an open-sourced way also helps you get your product’s usage patterns into the pre-training corpus of LLMs. (Author’s note: For pre-training text selection, AI labs are becoming much more discerning. So repetitive examples likely won’t move the needle there - you can brainstorm with an AI agent about the different potential use cases of your product, or dig into support tickets for inspiration.)

</div>

Multiple studies ([Liu et al. 2023](https://arxiv.org/abs/2307.03172), [Chroma 2025](https://research.trychroma.com/context-rot)) have confirmed the "Lost in the Middle" effect: LLMs process information at the start and end of a context window more reliably than information in the middle. Anthropic's [context engineering guide](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) also notes that long-running agents go through periodic "compaction", which is effectively summarizing their context to free up space, and the summarization process is also susceptible to this position bias.

**What to do**: Put the most important information first in every section ([Wu et al. 2025](https://arxiv.org/abs/2502.01951)). The LLM won’t benefit from gradual build up to the key point like humans do. If your API has required parameters, list them before optional ones. If there's a critical constraint, put it in the first few sentences, not the fourth paragraph (author’s aside: my perpetual struggle is I only think of the important assumptions after sitting down to write for a while).

Before:

<pre style="white-space: pre-wrap;">Our payment processing endpoint provides a flexible way to handle transactions. It supports multiple currencies and payment methods. Transactions are processed asynchronously and results are delivered via webhook. Note: all amounts must be in the smallest currency unit (cents for USD).</pre>

After:
<pre style="white-space: pre-wrap;">POST /payments
Processes a payment asynchronously. Amounts MUST be in smallest currency unit (cents for USD). Results delivered via webhook to configured endpoint. Supports multiple currencies and payment methods.</pre>

</div>

The 2025 "Context Rot" paper by [Chroma](https://research.trychroma.com/context-rot) found that LLM performance degrades as context grows, with measurable drops starting around 2,500 words and more significant degradation beyond 8,000. Obviously, this varies based on the model, but most of the research on this topic agrees that context longer than 25k words is not optimal.

Not surprisingly, going too short hurts too. [Zhang et al. 2025](https://arxiv.org/abs/2510.04618) showed that aggressively summarizing 13k words to under 500 causes a sudden drop in accuracy from 66.7% to 57.1%.

**What to do**: So what’s a sweet spot for the length of each retrievable section? It depends on how your docs are consumed. 

- For RAG systems, the research seems to suggest roughly **500–2,500 words for the chunk between headings** (This number might keep going up as the model gets better). 
- For coding agents reading files from disk, it's the whole file, but here it doesn’t matter as much because agents will use `grep` and `sed` to get around it. 
- For web search, the retrievable section is the page. So if you have very long pages containing multiple topics, consider splitting them. 

### 6. Tailor doc length to task complexity (advanced)

Finally, a caveat of chunk length is that research shows that the ideal agent context can depend on what the agent is trying to do with the information. [Zhang et al. 2025](https://arxiv.org/abs/2512.24601) found that more complex problems degrade at shorter context lengths than simpler ones. For example, simple needle-in-a-haystack lookups work fine even at 1M+ tokens, but multi-step reasoning tasks start failing much earlier (between 16k to 33k tokens).

This means a doc explaining a straightforward REST endpoint (look up parameters, make a call) can afford to be longer and more verbose than a doc explaining a complex multi-step workflow with conditional branching and state management. It’s a bit counterintuitive, yes. The hypothesized explanation is that the agent needs more token headroom for reasoning about the harder task.

**What to do**: Match your doc length to the task's reasoning complexity. Maybe think of it as a “reasoning density” that you want to keep relatively consistent across your doc pages, which means that for simple tasks (authentication, parameter references, error code tables), pack in as much relevant detail as you want into the page, but for pages with complex workflows, consider limiting to only one or two of these workflows per page. 

### 7. Keep your style guide and AGENTS.md separate (advanced)

This one only had single digit gains, so putting it as the last point, but this is probably the easiest one to implement. The research shows that when the context has task-level instructions (requires reasoning) and formatting requirements mixed up together, models often do worse on the actual task ([Deng et al. 2025](https://arxiv.org/abs/2510.03595)). 

**What to do**: Keep execution-critical guidance in your `AGENTS.md` or other agent instruction files; keep style guide as a separate file if you can.￼
