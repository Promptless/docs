Llms.txt has been around for a while, and we have not seen any definitive indication that the main AI agents are actually using it. The doc hosting providers, understandably, argue that these files have value, but more neutral parties on the SEO side — [Wix](https://www.wix.com/studio/ai-search-lab/llms-txt-myths), [SE Ranking](https://seranking.com/blog/llms-txt/), [Daydream](https://www.withdaydream.com/library/llms-txt-hype-vs-reality) — show from data across over 300,000 sites that it doesn't make a measurable difference. One might argue that SEO is different from agent support, but if these llms.txt pages aren't even getting indexed, they're most likely not making it into agent context either.

However, independent of llms.txt, it's undeniable that more agents are accessing docs and this trend will likely continue, to the point where the primary consumer will be agents, not humans. When that happens, patterns that worked best for humans may not work best for agents. This article outlines concrete ways you can tweak your docs for that potential reality.

## Why reformatting isn’t enough?

To learn to use your product, humans have to build a mental model. We connect new concepts to things we already know, and fill the gaps with intuition. Unfortunately, cognitive load theory (Sweller, 1988) shows that human working memory caps out at 7 items (or 2 if it's me before coffee). This is why good human docs use progressive disclosure by design.

LLMs don't build mental models. At inference time, the attention mechanism computes relevance scores between input tokens, performing a kind of soft retrieval over whatever's in the context window ([Vaswani et al. 2017](https://arxiv.org/abs/1706.03762)). When insufficient context is provided, the model falls back on its training to fill the gap, and that's where most hallucinations come from ([Kalai et al. 2025](https://arxiv.org/abs/2509.04664)).

## C - Chunk self-contained sections

Docs are being consumed by agents in different ways today: some are chunked and live in RAG systems (e.g. [Cursor](https://cursor.com/blog/secure-codebase-indexing)), some in file systems, and some via web search. If a lot of your docs traffic comes from QA bots or coding assistants, this section is especially relevant.

#### 1. Make each chunk more self-contained

When LLMs retrieve sections (whether full pages or the text between headings), you want to keep each section self-contained. You can even selectively repeat key context rather than relying on surrounding sections ([Anthropic](https://www.anthropic.com/engineering/contextual-retrieval), 2024; [Wang et al. 2025](https://aclanthology.org/2025.findings-acl.422/)). Avoid phrases like "see above," "same as before," etc., which break self-containment. Many writers we work with already have this in their style guide — worth mentioning here because agents suffer even more from these forward/backward references than human readers do.

Example — bad (depends on prior context):
```
## Refunds

Using the payment object from above, you can issue a refund 
by calling the refund endpoint described in the previous page.
```

Example — good (self-contained):
```
## Refunds: POST /refunds

Creates a refund for an existing payment.
Requires: Bearer token authentication (Authorization header).

Parameters:
  payment_id (string, required) — ID of the payment to refund 
    (format: "pay_" + 24 alphanumeric characters)
  amount (integer, optional) — Partial refund amount in smallest 
    currency unit. Defaults to full payment amount.
    Must not exceed original payment amount.
  reason (string, optional) — One of: "duplicate", "fraudulent", 
    "requested_by_customer"

Refund is processed against the original payment method.
Processing time: 5-10 business days depending on payment method.
```

#### 2. Chunk length matters

The 2025 "Context Rot" paper by [Chroma](https://research.trychroma.com/context-rot) shows that context degradation starts becoming measurable around 2,500 words, with significant drops beyond 8,000 words. Note that the exact thresholds vary by model. [Du et al. 2025](https://arxiv.org/abs/2510.05381) independently confirmed performance degradation at long context lengths (using open-source models), and a blog by [ElevenLabs](https://www.arcade.dev/blog/elevenlabs-agent-skills-paul-asjes-interview) echoes the same finding.

But shorter isn’t always better, another research paper ([Zhang et al. 2025](https://arxiv.org/abs/2510.04618)) pushes back and shows that if you try to summarize 13k words to less than 500, you also get performance degradation.

So there is a sweet spot. **Aim for 500-2500 words per retrievable section.**

## R - Raise critical info first

Multiple studies ([Liu et al. 2023](https://arxiv.org/abs/2307.03172), [Chroma 2025](https://research.trychroma.com/context-rot)) have confirmed the "Lost in the Middle" effect: information at the start and end of a context window is processed more reliably than information buried in the middle.

Long-running agents periodically go through a "compaction" process, in which salient points are preserved but low-priority execution details are dropped. Compaction is essentially summarization — and summarization is also susceptible to the "Lost in the Middle" effect ([Anthropic 2025](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)).

**What to do**: Put the most important information first in every section [Wu et al. 2025](https://arxiv.org/abs/2502.01951). State it immediately. The LLM won’t benefit from gradual build up to the key point like humans do. If your API has required parameters, list them before optional ones. If there's a critical constraint, put it in the first sentence, not the third paragraph (author’s aside: my perpetual struggle is I only think of the important assumption after sitting down to write for a while)

Before:

<pre style="white-space: pre-wrap;">Our payment processing endpoint provides a flexible way to handle transactions. It supports multiple currencies and payment methods. Transactions are processed asynchronously and results are delivered via webhook. Note: all amounts must be in the smallest currency unit (cents for USD).</pre>

After:
<pre style="white-space: pre-wrap;">POST /payments
Processes a payment asynchronously. Amounts MUST be in smallest currency unit (cents for USD). Results delivered via webhook to configured endpoint. Supports multiple currencies and payment methods.</pre>

## A - Abundant examples (many-shot + negative)

#### 1. More rich examples

Human docs typically show one or two examples per topic. However, research demonstrates significant performance gains going from few-shot to many-shot examples, with large numbers of examples sometimes outperforming fine-tuning a model ([Agarwal et al. 2024](https://arxiv.org/abs/2404.11018)).

Include negative examples aka "don't do this". LLMs learn as much from what a bad output looks like as from good ones ([Mo et al. 2024](https://arxiv.org/abs/2402.11254)).

**What to do**: Add more examples to your agent-facing docs, both positive and negative.

Example — what human docs provide:

```
curl -X POST /payments \
  -d '{"amount": 5000, "currency": "usd"}'
```

Example — what agent docs should provide:

```
## Examples: POST /payments

### Basic payment (USD)
Request: {"amount": 5000, "currency": "usd", "source": "tok_visa"}
Response (200): {"id": "pay_1a2b", "status": "processing", "amount": 5000}

### Payment with idempotency
Request: {"amount": 5000, "currency": "usd"}
Header: Idempotency-Key: "order_12345"
Response (200): {"id": "pay_1a2b", "status": "processing"}
Second call with same key → returns identical response, no new charge

### Multi-currency payment
Request: {"amount": 5000, "currency": "eur", "source": "tok_visa"}
Response (200): {"id": "pay_3c4d", "status": "processing", 
  "exchange_rate": 1.08, "settled_amount": 5400, "settled_currency": "usd"}

### Insufficient funds
Request: {"amount": 5000000, "currency": "usd", "source": "tok_declined"}
Response (402): {"error": {"code": "card_declined", 
  "decline_code": "insufficient_funds", "message": "..."}}

### Invalid currency code
Request: {"amount": 5000, "currency": "xyz"}
Response (400): {"error": {"code": "invalid_currency", 
  "param": "currency", "message": "..."}}
```

#### 2. Edge cases

Listing every edge case in human-facing docs would clutter the reading experience, so we often decide to exclude them and leave them to support. But when an agent encounters an undocumented edge case, it doesn't just shrug and move on. It hallucinates a solution.

A recent paper studied automated prompt optimization for LLM agents ([Brookes et al. 2025](https://arxiv.org/abs/2512.09108)). They found that many prompts contained vague guidance like “consider edge cases” or “be thorough.” After rewriting these into explicit, structured instructions — breaking tasks into substeps, specifying verification procedures, and making reasoning requirements concrete — accuracy improved by 10.1%–36.9%.

**What to do**: Don't shy away from documenting edge cases for your critical products. Use structured formats that agents can pattern-match against.

Example:

```
## Edge Cases: POST /payments

- Amount of 0: Returns 400. Zero-amount transactions not supported.
- Currency mismatch: If currency doesn't match account default, 
  conversion applied automatically. Rate available in response header 
  X-Exchange-Rate.
- Duplicate idempotency key within 24h: Returns original response, 
  no new transaction created. After 24h, key is released.
- Amount exceeding per-transaction limit ($50,000 USD equivalent): 
  Returns 403 with error code AMOUNT_LIMIT_EXCEEDED. Cannot be 
  overridden via API; requires dashboard configuration.
- Webhook endpoint returning non-2xx: Retried 3 times with 
  exponential backoff (1min, 5min, 30min). After 3 failures, 
  event stored in dead letter queue for 7 days.
```

## F - Failure catalogs & known bugs

#### 1. Document known bugs and workarounds explicitly

In human docs, issues and bugs live in release notes, changelogs, GitHub issues, or nowhere at all. Agents don't always know where to go for this information. If a bug isn't in their context, they'll generate code that hits it and have no idea why it fails.

Research in coding agents ([Dong et al. 2025](https://arxiv.org/pdf/2508.00083)) shows that agents struggle the most with "unwritten team conventions" and "non-public, highly contextualized information”. If these details are not exposing any trade secrets, putting them into your docs can help agents a lot.

**What to do**: Add a known issues section to your agent docs. Structure each issue with the trigger condition, the symptom, and the workaround.

Example:

```
## Known Issues: POST /payments

- ISSUE: Webhook delivery fails silently for URLs with trailing 
  slashes on certain CDN configurations.
  TRIGGER: webhook_url ends with "/"
  WORKAROUND: Strip trailing slash from webhook URL before submitting.
  STATUS: Fix planned for v2.4 (Q2 2026)

- ISSUE: Idempotency keys containing "+" character are URL-decoded 
  to spaces, causing key mismatch on retry.
  TRIGGER: Idempotency-Key header contains "+"
  WORKAROUND: URL-encode the key, or avoid "+" characters.
  STATUS: Won't fix (documented behavior)
```

#### 2. Include failure modes

Research on agent self-recovery (Del Rosario et al. 2025) found that agents with access to structured failure information performed significantly better than agents that reason about errors from a fuzzy “common sense”. The latter agents can get stuck in a failure loop and not know how to troubleshoot.

**What to do**: Provide a structured error catalog that maps error codes or messages to causes, and what the recovery action should be.

Example:

```
## Error Catalog: POST /payments

| Status | Code                  | Retryable | Recovery                          |
|--------|-----------------------|-----------|-----------------------------------|
| 400    | invalid_amount        | No        | Check amount > 0, in minor units  |
| 400    | invalid_currency      | No        | Use GET /currencies for valid list |
| 401    | auth_failed           | No        | Regenerate API key from dashboard  |
| 402    | card_declined         | No        | Prompt user for different method   |
| 409    | idempotency_conflict  | No        | Use new idempotency key            |
| 429    | rate_limited          | Yes       | Retry after Retry-After header (s) |
| 500    | internal_error        | Yes       | Retry with exponential backoff     |
| 503    | service_unavailable   | Yes       | Retry after 30s, max 3 attempts    |
```

## T — Tailor doc length to task complexity
Finally, a caveat of chunk length is that research shows that the ideal agent context can depend on what the agent is trying to do with the information. The Recursive Language Models [paper](https://arxiv.org/abs/2512.24601) (late 2025) makes this explicit: More complex problems degrade at shorter lengths than simpler ones. Simple needle-in-a-haystack lookups work fine even at 1M+ tokens, but multi-step reasoning tasks start failing much earlier.

This means a doc explaining a straightforward REST endpoint (look up parameters, make a call) can afford to be longer and more detailed than a doc explaining a complex multi-step workflow with conditional branching and state management. The agent needs more cognitive headroom for the harder task, so the surrounding documentation needs to take up less of it.

**What to do**: Match your doc density to the task's reasoning complexity. For simple lookups (authentication, parameter references, error code tables), pack in as much relevant detail as you want — the agent is just retrieving, not reasoning. For complex workflows (multi-step integrations, conditional logic, state management), keep each section tighter and break the workflow into smaller, independently retrievable steps. The agent needs context window headroom to reason through the hard parts.
